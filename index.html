The frame

Context sprawl is the next wall, not another training program.

Roughly 220 AI solutions in production, 140 digital employees operating with human managers, 20,000 Empowered Builders on the platform, a model-agnostic substrate routing between OpenAI, Google Cloud, and internal models.

The problem that shows up next is context sprawl, interpretability drift, and audit load that grows faster than the human attention available to track it.

The operating lens we use with every engagement is a 60/30/10 split. Roughly 60% of what looks like an AI problem is traditional code and database work. 30% is rule-based logic. 10% is a genuine agent problem.

Inside Eliza the ratio shifts, because the platform is already doing the heavy lifting. The question at the team level stays the same: when a builder reaches for an agent, is the right layer chosen for the right piece of the workflow.

Most "agent sprawl" is not an agent problem. It is context that was never given a proper home. The methodology below gives context a home.

Proof points

A published methodology and a regulated-enterprise cohort that shipped it.

Methodology

Interpretable Context Methodology (ICM)

Folder structure as agent architecture. Layered context from L0 identity through L4 working artifacts, with measurable interpretability and reproducibility gains. Submitted to ACM TiiS. 52-member practitioner community. U-shaped edit pattern observed in real cohorts.

The paper is the one BNY would actually want. The bank's agentic build-out is about to hit the governance wall that ICM was designed to address: making agent context legible, versionable, and auditable across thousands of builders.
